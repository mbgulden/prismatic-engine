## 1. The Problem

Multiple AI agents (Fred, Kai, AGY, Jules) editing the same GitHub repository on the same staging branch (`deploy-fresh`) without stepping on each other's work.

**Specific threats:**
- Content agent (Kai) and code agent (Fred) modifying different files but pushing to same branch
- Two agents editing the same file concurrently → merge conflict or silent overwrite
- Agent A doesn't know Agent B changed a shared dependency
- No attribution in git history for which agent made which change
- Force-push destroying another agent's work

**This is a problem every multi-agent system hits.** No mainstream solution exists.

### Agent Contract Interface

```typescript
interface AgentContract {
  threadId: string;
  role: string;
  taskDescription: string;
  allowedDirectories: string[];       // Lane assignment
  readOnlyDirectories: string[];      // Read-only lanes
  targetHead: 'Antigravity UI' | 'Headless API' | 'Local AI' | 'GitHub Jules';
  budgetLimit?: number;
  localContextMax?: number;
}
```

### Enforcement Prompt (generated by ContractManager)
```
HARD CONSTRAINTS:
1. RESTRICTED SCOPE: You may only edit files within allowedDirectories
2. READ-ONLY SCOPE: Refer to readOnlyDirectories for context only
3. MUTEX LOCKS: Run `.antigravity/swarm.js lock <filepath> <agentId>` before editing
4. Release locks when done: `.antigravity/swarm.js unlock <filepath> <agentId>`
5. Signal completion by deleting your contract file
```

## 6. Proposed Architecture

```
Prismatic Engine
│
├── 1. Lane Assignment (WORKSPACE_GOVERNANCE.md)
│   ├── Kai → content/*, active-oahu/*
│   ├── Fred → src/, infra/, deploy/
│   ├── AGY → assets/, designs/, research/
│   └── Jules → PR creation, code review
│
├── 2. File Claim System (swarm_locks.json)
│   ├── Existing SwarmLockManager — works
│   └── Add: heartbeat timeout, directory-level locks
│
├── 3. Staging Branch Governor
│   ├── Content agents → feature branches (content/*)
│   ├── Code agents → feature branches (feature/*)
│   ├── Only Fred merges to deploy-fresh
│   └── Pre-push hooks validate ownership
│
├── 4. Agent Identity in Git
│   ├── Every commit: "[Kai] description" prefix
│   ├── git config user.name per agent profile
│   └── Pre-push hook: "Do you own these files?"
│
├── 5. Conflict Predictor
│   ├── Pre-push: diff HEAD against staging
│   ├── If another agent touched same file → BLOCK
│   └── Report: "Fred modified src/kadence.css — reconcile first"
│
└── 6. Heartbeat + Grey Release
    ├── Each agent pings lock registry every 60s
    ├── Dead agent → auto-release its locks
    └── Staging always reflects last clean state
```

## 7. Key Insight

The Antigravity Orchestration Hub already has **90% of the Prismatic Engine** — it just needs:
1. The Hermes adaptation layer (agents loaded via Hermes profiles, not VS Code)
2. The branch governance protocol (content vs feature branches)
3. The pre-push hooks for conflict prediction
4. The heartbeat system for dead lock release

The `SwarmLockManager`, `ContractManager` (with `allowedDirectories`/`readOnlyDirectories`), and `HandoffProtocol` are complete and tested. The `swarm.js` CLI is deployable immediately. The `swarm_locks.json` data store is in place.

This is not a design problem. It's an **integration and convention** problem.
