# Soul Alignment Guide

*Every feature, every line of code, every decision. Test it against the soul before you build.*

---

## The Litmus Test (6 Questions)

Before building any feature, ask:

1. **Does this move an idea closer to reality?** If it produces a plan instead of the thing itself, it's incomplete.

2. **Does this help the team work together?** A feature that isolates agents or creates handoff friction is working *against* the soul.

3. **Can I walk away and trust it?** If the feature requires me to babysit it, it's not ready.

4. **Is this the whole spectrum or just one color?** A feature that only produces one output type (just code, just design, just text) when the goal needs many is a fragment, not a solution.

5. **Does this respect my standards?** If it generates output without my voice, taste, or brand guidelines, it's not aligned.

6. **Does this make the next thing faster?** Every feature should compound. If it only solves today's problem without making tomorrow's problem easier, it's tactical, not strategic.

---

## Soul Violations (Red Flags)

| Red Flag | What It Actually Means | Fix |
|----------|----------------------|-----|
| "I'll just write a spec first" | You're deferring reality. Skip to the prototype. | Write the spec AND the implementation in the same session. |
| "Let's add a dashboard for that" | You're building visibility when you should be building autonomy. | Does the agent need a dashboard, or does it need to not require oversight? |
| "We need to discuss scope" | The agents don't trust each other yet. | Strengthen lane conventions, not meeting protocols. |
| "Let me write a plan" | You're producing intermediate artifacts instead of final ones. | The plan is the output. If the output is a plan, you're not done. |
| "That's Phase 5" | You're deferring the hard, valuable part. | Ship the simplest version of the hard part now. Polish later. |

## Soul Alignment by Phase

### Phase 1 — Convention
**Soul question:** *Do the agents know their lanes and trust each other?*
**Alignment:** A team that doesn't collide doesn't need locks. The conventions ARE the coordination.

### Phase 2 — Locking
**Soul question:** *Did collisions actually happen?*
**Alignment:** Build the fence at the cliff edge, not on flat ground. Only implement when proven necessary.

### Phase 3 — Hooks
**Soul question:** *Can I trust that quality is enforced without me checking?*
**Alignment:** Hooks are the substitute for my oversight. They should be invisible unless something breaks.

### Phase 4 — Visibility
**Soul question:** *When I check in, can I see the truth instantly?*
**Alignment:** The dashboard should answer "is the team working?" in one glance. Not a data dump — a pulse check.

### Phase 5 — The Loop
**Soul question:** *Can I set it and truly forget it?*
**Alignment:** The 7-step loop is the engine's autonomous nervous system. It should feel like magic, not machinery.

### Phase ∞ — The Dream
**Soul question:** *Can I have a business idea at 8 AM and hold the evidence at 5 PM?*
**Alignment:** This is the bar. Everything before it is infrastructure for this moment.

## How to Infuse Soul Into Deliverables

### Code
- Every module has a docstring that explains *why it exists*, not just *what it does*
- Error messages say what happened AND what the soul expects next
- Configuration files have descriptions that explain intent

### Reports
- Every report starts with the "so what" — how does this move ideas toward reality?
- Technical specs include a "soul alignment" section
- Gap analyses prioritize based on distance from reality, not technical debt

### Tasks
- Every Linear task has a "why this matters" field (not just acceptance criteria)
- Priority is determined by: does this get us closer to the 8AM→5PM dream?
- Assignments consider the full spectrum — who should collaborate, not just who should execute

### Communication
- We talk about *outcomes*, not features. "This lets you walk away" not "this adds a lock manager."
- We celebrate *results in the world*, not code shipped. "Business cards arrived" not "PR merged."
- We measure by *reality gap closed*, not lines of code written.

## The Ultimate Test

In one year, what does success look like?

**The wrong answer:** "The Prismatic Engine has a mature plugin ecosystem, comprehensive documentation, and 100+ GitHub stars."

**The right answer:** "Someone with no technical background woke up with an idea, opened Prismatic Engine, described it, and by the end of the day had a registered business, a live website, and their first customer. They didn't write a line of code. They didn't file a single form. They just *thought* it, and it *happened*."

That's the soul. Measure everything against it.
